Types of Testing

The lines between different types of testing can be a little blurry. Different companies and teams will have slightly different definitions for each type. This is a general guide.

Unit

Test that the individual parts of a program work as expected. The vast majority of tests in an application's suite will be of type "unit."

  • Tests the smallest possible pieces of programmatic functionality, concentrating on inputs and outputs.
  • One failed test should correspond with one defect in the code.
  • These run in a development environment.
  • They're run multiple times while writing new code, and before new code is merged to main branch.
  • Primary stakeholders are programmers.

Integration

Test that the pieces of the application work together as expected. This includes external dependencies like APIs and databases, when possible. There are fewer of these types of tests.

  • Test more than one piece of functionality in concert.
  • A failure in an integration test points to a problem in the interface between sections of code. Sometimes, there are false failures due to things like network connectivity.
  • These run in a development environment.
  • They're run a few times while writing new code, and before new code is merged to main branch.
  • Primary stakeholders are programmers.

Acceptance

Test that the application provides the expected user experience, as defined by the product owner. There are very few of these types of tests.

  • Test an entire user flow (e.g. signing up for a new account).
  • A failure does not clearly indicate where the code defect lies.
  • These run in a production-like environment.
  • They are run before new code is merged to main branch, and after every deploy.
  • Primary stakeholder is product owner.

Performance

Test a system's performance characteristics, like page load time or response latency.

Load

Test how performance holds up under heavy traffic. This may be considered a subset of performance or acceptance testing.
